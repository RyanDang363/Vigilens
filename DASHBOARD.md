# Dashboard Plan

## Purpose

A manager-facing web dashboard that displays employee performance reports generated by the multi-agent system. The dashboard is the primary way managers review health/safety findings, efficiency findings, and training recommendations produced from video analysis.

It is **not** responsible for running agents, processing video, or making decisions — it consumes the structured output that the orchestrator produces.

## Users

- **Manager** — views reports, reviews employee profiles, sees infraction history
- Employees do not access the dashboard directly; they receive training reports via email (handled by the browser agent)

## Core Features

### 1. Authentication

- OAuth login (Google OAuth is the simplest for hackathon)
- Only authenticated managers can access the dashboard
- No employee self-service for v1

### 2. Employee Roster

Each employee has a profile card showing:

| Field | Notes |
| --- | --- |
| Name | Display name |
| Employee ID | Internal identifier |
| Role / Station | e.g. "Prep Cook - Station 2" |
| Start Date | When they joined |
| Total Findings | Count of all findings across clips |
| Highest Severity | The worst finding on record |

**No DOB or SSN** — unnecessary for the demo and creates privacy risk.

### 3. Report View (per employee)

The main view after clicking an employee. Shows all findings from the most recent evaluation (or a selected clip/session).

Each finding card displays:

| Field | Source |
| --- | --- |
| Finding Type | `concluded_type` from Health/Efficiency Agent (e.g. "Cross Contamination") |
| Status | `confirmed_violation`, `possible_violation`, `insufficient_evidence` |
| Severity | Color-coded badge: low (gray), medium (yellow), high (orange), critical (red) |
| Finding Class | `code_backed_food_safety` or `workplace_safety_rule` or `efficiency` |
| Policy Citation | Code + section number (e.g. "FDA Food Code 2022 - 3-302.11") |
| Rule Summary | `short_rule` from the policy reference |
| Timestamp | Start - end time in the clip |
| Video Clip | Embedded clip player or link to the relevant segment |
| Confidence | Confidence score from the vision pipeline |
| Coaching | `training_recommendation` text |
| Source URL | Link to the official policy document |

### 4. Report Summary

At the top of each report:

- Total findings count
- Breakdown: code-backed vs guidance vs efficiency
- Highest severity
- Quick stat bar: e.g. "3 health findings, 1 efficiency finding, highest: HIGH"

### 5. Infraction Timeline

A timeline view showing all findings for an employee across multiple clips/sessions, ordered chronologically. Helps managers spot patterns (e.g. repeated handwashing issues across shifts).

### 6. Email Integration

- "Send Report" button on the report view
- Triggers the browser agent to email a formatted report to the employee and/or manager
- Report includes: findings summary, top coaching recommendations, links to training docs

```text
|                                                          |

```

### Visual Style

- Clean, professional — think internal operations tool, not consumer app
- Severity color coding throughout:
  - `low` — gray
  - `medium` — yellow/amber
  - `high` — orange
  - `critical` — red
- Finding class badges: "Health Code" (blue), "Workplace Safety" (purple), "Efficiency" (teal)
- Cards-based layout for findings — each finding is a distinct card
- Dark/light mode not needed for v1

### Key Pages

1. **Login** — OAuth flow
2. **Roster** — grid/list of all employees with summary stats
3. **Employee Report** — findings for a specific employee + summary + timeline
4. **Session Detail** — deep dive into a single clip/session evaluation

## Tech Stack

| Layer | Choice | Reason |
| --- | --- | --- |
| Frontend | React (Next.js) or plain React + Vite | Fast to build, component-based, good for cards/tables |
| Styling | Tailwind CSS | Utility-first, fast iteration for hackathon |
| Auth | NextAuth.js or Firebase Auth (Google OAuth) | Quick OAuth setup |
| Backend API | FastAPI (Python) | Already in Python ecosystem with the agents |

| Deployment | Localhost for demo | No need to deploy for hackathon |

## Data Flow — How the Dashboard Gets Its Data

```text
Video -> TwelveLabs -> Observations
                          |
                          v
                    Orchestrator Agent
                     /            \
                    v              v
             Health Agent    Efficiency Agent
                     \            /
                      v          v
                    Orchestrator Agent
                          |
                          v
                   Backend API (FastAPI)
                          |
                          v
                     Database (SQLite)
                          |
                          v
                     Dashboard (React)
```

1. The orchestrator collects findings from Health + Efficiency agents
2. The orchestrator sends the combined report to the backend API
3. The backend stores it in the database, keyed by employee_id, clip_id, and session
4. The dashboard fetches reports from the backend API and renders them

### API Endpoints (FastAPI)

```text
POST /api/reports              — orchestrator submits a completed report
GET  /api/employees            — list all employees with summary stats
GET  /api/employees/{id}       — single employee profile
GET  /api/employees/{id}/reports — all reports for an employee
GET  /api/reports/{report_id}  — single report with all findings
POST /api/reports/{id}/email   — trigger the browser agent to email the report
```

## Integration with Agents

### From Health Agent

The dashboard consumes `HealthEvalResponse` which contains:
- `findings[]` — each with `concluded_type`, `status`, `severity`, `policy_reference`, `training_recommendation`
- `code_backed_count`, `guidance_count`, `highest_severity`

### From Efficiency Agent

Same structure — the efficiency agent will produce findings with:
- `concluded_type` like `phone_usage`, `extended_chatting`, `slow_task_execution`
- `finding_class = "efficiency"`
- Its own coaching recommendations

### From Orchestrator

The orchestrator merges Health + Efficiency findings into a single report and POSTs it to the backend API. The dashboard never talks to agents directly.

### From Browser Agent

The browser agent handles the "Send Report" email action. When a manager clicks the button, the dashboard calls `POST /api/reports/{id}/email`, the backend sends a message to the browser agent, and the browser agent composes and sends the email.

## Implementation Order

1. **Backend API** — FastAPI with SQLite, define the endpoints and schema
2. **Database** — create tables, seed with mock employee data
3. **Frontend scaffold** — React app with routing (Login, Roster, Report pages)
4. **Auth** — Google OAuth via NextAuth or Firebase
5. **Roster page** — fetch and display employees
6. **Report page** — fetch and display findings as cards with severity badges
7. **Integration** — wire the orchestrator to POST reports to the backend
8. **Email button** — trigger the browser agent from the report page

## What to Skip for v1

- Real-time updates / WebSockets — just refresh
- Dark mode
- Export to PDF
- Complex filtering/search
- Employee self-service
- Historical trend charts (timeline is enough)
- Mobile responsiveness (demo on a laptop)
